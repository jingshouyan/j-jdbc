j-jdbc 是什么？
让数据库操作变简单

你是否曾经用 Java 写过数据库代码？如果你写过，你一定经历过这个循环：打开连接 → 创建语句 → 执行查询 → 处理结果集 → 关闭连接 → 关闭语句 → 捕获异常……重复几十次。更糟糕的是，你的代码里到处都是 SQL 字符串，拼错一个字就会在运行时崩溃。

j-jdbc一个基于 Spring JDBC Template 的 CRUD 增强框架，通过注解自动生成 SQL 语句，大幅减少数据库操作的样板代码。 就是为了解决这个问题而生的。它是一个轻量级的 Java 库，工作方式是："你告诉我数据结构，我来搞定数据库。"

「声明式」的魔法

j-jdbc 的核心思想是 声明式编程：你不用告诉代码"怎么"做（写 SQL），你只需要声明"是什么"（用注解描述数据）。框架会自动生成所有 SQL。

想象一下：你创建一个普通的 Java 类，加上几个注解，然后——嘭！——完整的增删改查、分页查询、条件过滤全部自动可用。不需要写一行 SQL。

🧩 隐喻：智能家居

传统 JDBC 编程就像手动操作家里每一个电灯开关——你得知道每个开关在哪、怎么按、哪个控制哪个房间。j-jdbc 就像一个智能家居系统：你只需要告诉它"客厅需要灯光"（声明一个带注解的对象），它就会自动帮你打开正确的开关、调节亮度、甚至在日落时自动亮起（自动生成 SQL、管理连接、处理分页）。

一行代码，一个世界

这是 j-jdbc 最令人惊叹的地方：你只需要 4 个文件 就能完成一个完整的数据库操作。

java // 1️⃣ 数据库对象 — 加注解描述表结构 @Table(value = "DEMO_USER", comment = "用户表") public class UserDO extends BaseDO { @Key @Column(comment = "主键") private String id; @Column private Integer age; @Column(value = "NICK_NAME_TT4") private String nickname; } // 2️⃣ DAO 接口 — 就一行！ public interface UserDao extends BaseDao<UserDO> {} // 3️⃣ DAO 实现 — 也是一行！ @Repository public class UserDaoImpl extends BaseDaoImpl<UserDO> implements UserDao {} // 4️⃣ 使用 — 直接调用 userDao.insert(user); // 插入 userDao.update(user); // 更新 userDao.find("U123@abc"); // 按主键查询 userDao.query(conditions); // 条件查询 userDao.delete("U123@abc"); // 删除

看到没？没有 SQL，没有连接管理，没有资源释放。 所有的数据库细节都被封装在 `BaseDaoImpl` 中。你可以把精力放在业务逻辑上，而不是重复编写 CRUD 代码。

用注解说清数据
对象到数据库的映射

如果说 j-jdbc 是一门语言，那么 注解（Annotation） 就是它的词汇。你不需要学习 SQL 的语法，也不需要配置 XML 映射文件——只需要在 Java 类上贴标签，框架就知道你想怎么存储数据。

想象一下：你有一个"用户"的概念，它有名字、年龄、标签。在数据库中，它是一个表；在 Java 中，它是一个类。j-jdbc 的注解就是连接这两个世界的桥梁。

@Table — 给类穿上数据库的外衣

@Table表注解。加在类上，告诉 j-jdbc 这个 Java 类对应数据库中的哪张表。如果不指定表名，默认使用类名。 是最顶层的注解。它告诉框架："嘿，这个 Java 类对应一张数据库表。"

java @Table( value = "DEMO_USER", // 数据库中实际的表名 comment = "用户表" // 建表时用的备注 ) public class UserDO extends BaseDO { // 字段... }
☕ Java 代码 中 文 解 释
@Table(value = "DEMO_USER")
"这个类对应数据库中的 DEMO_USER 表"
@Table(comment = "用户表")
"给这个表加个备注：用户表"
public class UserDO
UserDO 就是这张表的 Java 化身
🤯 啊哈！时刻

注意看 "Table" 注解的拼写： 它没有写成标准的 "Table"，而是使用了正确的拼写 "Table"。开发者通过这种一致性设计告诉我们：注解的命名可以直接映射到其作用——要让名称清晰到不需要文档也能猜到用法。

@Column — 精确控制每一列

@Column列注解。加在字段上，定义数据库列名、长度、类型、是否加密、是否 JSON 存储等属性。如果不指定，默认使用字段名作为列名。 是 j-jdbc 最强大的注解。它不仅能控制列名和长度，还能指定加密方式、JSON 序列化、数据类型推断等高级功能。

java // 简单用法 — 列名等于字段名 @Column private Integer age; // 指定列名和备注 @Column(value = "NICK_NAME_TT4", comment = "昵称") private String nickname; // JSON 格式存储 + 加密 @Column(json = true, encryptType = EncryptType.FIXED, encryptKey = "abcd", length = 1000) private List<String> tags; // 不可变字段（创建后不能改） @Column(immutable = true, order = 1001, comment = "创建时间") private Long createdAt;

数据类型自动推断： j-jdbc 会查看字段的 Java 类型，自动选择最合适的 MySQL 数据类型。`int` → INT, `String` 且长度小于 5000 → VARCHAR, `String` 且长度 ≥ 5000 → TEXT, `BigDecimal` → DECIMAL, `LocalDateTime` → DATETIME, `List` 且 `json=true` → JSON……你不用记住这些映射，框架会帮你处理。

@Key — 主键 & @Index — 索引

java // 主键 — 只能是 Long 或 String @Key(generatorIfNotSet = true) // 如果没设置值，自动生成 @Column(comment = "主键") private String id; // 单字段索引 @Index private Integer age; // 复合索引（类级别） @Index({"age", "nickname"}) @Index({"id", "name"}) public class UserDO ...

主键（@Key） 是每个表的灵魂。j-jdbc 要求主键只能是 Long 或 String 类型。如果设置了 `generatorIfNotSet = true`，当你插入数据时没有设置主键值，框架会自动生成一个唯一 ID。

索引（@Index） 可以在字段上（单字段索引）或类上（复合索引）使用。数据库索引就像是书的目录——它让查询速度从"逐页翻找"变成"直接翻到正确的页码"。在第 4 课中，我们会详细讲解主键是如何生成的。

幕后引擎：TableInfo 是如何工作的

所有的注解魔法都归结于一个核心类——TableInfo。它读取类上的所有注解、字段、以及父类的字段，构建出一张完整的"元数据地图"。

☕ TableInfo 构造函数 它做了三件事
tableName = clazz.getSimpleName();
1️⃣ 先以类名作为默认表名
@Table table = clazz.getAnnotation(Table.class);
2️⃣ 检查是否有 @Table 注解 → 有的话覆盖表名
Field[] fields = c.getDeclaredFields();
for (Field field : fields) {
ColumnInfo beanColumn = new ColumnInfo(field);
addColumnInfo(beanColumn);
}
3️⃣ 遍历所有字段→每个字段变成 ColumnInfo 对象→添加到地图中

注意：它会遍历整个类继承树（从当前类一直往上到 Object），确保父类如 BaseDO 中的 createdAt、updatedAt、deletedAt 字段也被纳入。

数据的旅程
CRUD 的生命之旅

现在你已经知道了 j-jdbc 用注解来描述数据。但当你调用 userDao.insert(user) 时——按下键盘到数据落库，中间发生了什么？

这一章我们将追踪一条数据从 Java 代码到数据库的完整旅程。你会发现 j-jdbc 的 SQL 生成器是真正的幕后英雄——它把简单的 Java 方法调用，翻译成精确的 SQL 语句。

🧩 隐喻：外卖配送

调用 userDao.insert(user) 就像在手机上点外卖。你（代码）只做了一件事——点了一下"下单"按钮。BaseDaoImpl 是外卖平台，接收你的订单。SqlGenerator 是厨房里的厨师，根据订单（注解信息）准备餐食（生成 SQL）。NamedParameterJdbcTemplate 是骑手，把餐食送到你的数据库门口。

一次查询的完整旅程

当你调用 userDao.find("U123@abc")，从方法调用到数据库查询结果返回，经历了这些步骤：

java — BaseDaoImpl.find() @Override public Optional<T> find(Object id) { // 1. 检查ID不为空 Preconditions.checkNotNull(id, "id is null"); // 2. 用 ConditionUtil 构建查询条件 List<Condition> conditions = ConditionUtil .newInstance() .field(key()).eq(id) .conditions(); // 3. 调用 queryField → SqlGenerator → 生成 SQL List<T> ts = queryField(conditions, null); // 4. 返回第一个结果 return ts.stream().findFirst(); }

ConditionUtil — 用 Java 写 WHERE 子句

ConditionUtil条件查询构建工具。提供链式调用 API 来构造 SQL 的 WHERE 条件，支持 eq、ne、gt、lt、gte、lte、like、in、notIn、between、empty 等操作。 是 j-jdbc 中最常用的工具之一。它让你用链式调用的方式构建查询条件，而不用手写 SQL。

☕ Java 条件代码 生 成 的 SQL
ConditionUtil.newInstance()
.field("age").gt(20).lte(89)
.field("nickname").like("%张三%")
.conditions()
WHERE age > 20 AND age <= 89
AND nickname LIKE '%张三%'

支持的所有条件操作：

.eq(val)等于 =
.ne(val)不等于 <>
.gt(val)大于 >
.lt(val)小于 <
.gte(val)大于等于 >=
.lte(val)小于等于 <=
.like(str)LIKE 模糊匹配
.in(list)IN 列表包含
.notIn(list)NOT IN 排除
.between(a,b)BETWEEN 范围
.empty(bool)IS NULL / IS NOT NULL

SQL 生成器 — Java 到 SQL 的自动翻译

j-jdbc 的 SQL 生成器体系由 SqlGenerator 接口定义，AbstractSqlGenerator 提供通用实现，SqlGenerator4Mysql 提供 MySQL 特化版本。这种设计让 j-jdbc 可以轻松支持其他数据库——只需要写一个新的实现类。

java — SqlGenerator 接口 public interface SqlGenerator<T> { SqlPrepared query(...); // SELECT 查询 SqlPrepared queryLimit(...); // 分页查询（带 LIMIT） SqlPrepared count(...); // SELECT COUNT(*) SqlPrepared insert(...); // INSERT SqlPrepared update(...); // UPDATE SqlPrepared delete(...); // DELETE List<SqlPrepared> createTableSql(); // CREATE TABLE SqlPrepared dropTableSql(); // DROP TABLE SqlPrepared addColumn(...); // ALTER TABLE ADD COLUMN String doc(); // 生成 Markdown 文档 }

看那个 where() 方法是如何将 Condition 对象翻译成 SQL 条件的——它遍历每个条件，检查哪个字段被设置了（eq、gt、like 等），然后拼装对应的 SQL 片段。这就像是一个精心设计的状态机！

🤯 啊哈！时刻

「参数化查询」是 j-jdbc 默认的安全保障： 所有的 SQL 参数都使用命名参数（如 :id__eq）而不是拼字符串。这意味着 SQL 注入攻击从一开始就被杜绝了——你不需要额外做任何安全配置。这是框架设计者的明智决定。

插入与更新 — 智能的 null 处理

j-jdbc 有一个非常贴心的设计：null 值不插入，null 值不更新。这意味着你可以只设置你想操作的字段，其他字段保持不变。

java — 智能 null // 插入时：只有设置了值的字段会出现在 INSERT 语句中 UserDO user = new UserDO(); user.setName("Alice"); user.setAge(25); // tags 为 null → 不会出现在 INSERT 中 userDao.insert(user); // → INSERT INTO DEMO_USER (id, name, age) VALUES (..., :name, :age) // 更新时：只有非 null 字段会更新 user.setAge(26); userDao.update(user); // → UPDATE DEMO_USER SET age=:age__set WHERE id=:id__eq // name 不会被更新！

更酷的是，如果你确实想把某个字段更新为 null，可以实现 updateNullFields() 方法告诉框架："这个字段我不是忘了设置，我是想把它清空。"

隐藏的超能力
加密、ID 生成与安全删除

如果说前面三章是 j-jdbc 的"基础技能"，那这一章就是它的隐藏超能力。很多 CRUD 框架只能做增删改查，但 j-jdbc 内置了几个大多数开发者需要但懒得实现的特性——字段加密、分布式 ID 生成、逻辑删除和事件通知。

作为直觉型编程者，了解这些能力极其有价值：当你的 AI 助手建议在数据库中存储敏感数据时，你可以说"用 j-jdbc 的字段加密，别存明文"——这听起来就像你是个真正的后端架构师。

🔐 字段级加密 — 透明保护

字段加密在数据写入数据库时自动加密指定字段，读取时自动解密。对应用程序代码完全透明——你读写的是明文，数据库里存的是密文。 意味着你可以在 Java 代码中正常读写数据，但数据库里存的是密文。即使数据库被拖库，黑客看到的也是一堆乱码。

java // ===== FIXED 加密 ===== // 用固定密钥 "abcd" 加密。查询时框架自动加密比较。 @Column(encryptType = EncryptType.FIXED, encryptKey = "abcd") private String encryptTest; // ✓ 支持 eq, neq, in, notIn 查询 // ✗ 不支持 like 模糊查询 // ===== FIELD 加密 ===== // 用另一个字段 "id" 的值作为加密密钥 @Column(encryptType = EncryptType.FLIED, encryptKey = "id") private String name; // ✗ 每条记录密钥不同，不能作为查询条件 // ✓ 更安全 —— 泄露一条不影响其他数据
☕ 加密方式选择 什 么 时 候 用
EncryptType.FIXED
固定密钥：需要作为查询条件的字段（如手机号、邮箱），所有记录使用同一个密钥加密
EncryptType.FLIED
属性密钥：不需要查询的敏感字段（如密码、地址），每条记录使用不同的密钥（用其他字段的值加密）
EncryptType.NONE
不加密：默认选项

底层加密：AES 算法

j-jdbc 默认使用 AES（高级加密标准）来加密数据。AES 是目前最广泛使用的对称加密算法——从银行系统到 WhatsApp 都在用它。

java — Aes.java public class Aes implements Encryption { // 加密流程： public String encrypt(String content, String password) { // 1. 用密码生成加密密钥 KeyGenerator keyGen = KeyGenerator.getInstance("Aes"); SecureRandom random = SecureRandom .getInstance("SHA1PRNG"); random.setSeed(password.getBytes("utf-8")); keyGen.init(128, random); // 2. 初始化密码器（加密模式） Cipher cipher = Cipher.getInstance("Aes"); cipher.init(Cipher.ENCRYPT_MODE, secretKey); // 3. 加密 & Base64 编码 byte[] result = cipher.doFinal(content.getBytes("utf-8")); return Base64.getEncoder().encodeToString(result); } }

你也可以通过 EncryptionProvider 设置自己的加密算法——只需实现 Encryption 接口（encrypt/decrypt 两个方法），然后调用 EncryptionProvider.setEncryption(yourImpl)。

❄️ 雪花 ID — 分布式系统的身份证

雪花算法（Snowflake）Twitter 开源的一种分布式 ID 生成算法，生成 64 位的唯一整数。结构：时间戳（41位）+ 工作节点（10位）+ 序列号（12位）。可以在不依赖数据库的情况下生成全局唯一 ID。支持 1024 个节点，每毫秒单节点生成 4096 个 ID。 是 Twitter 发明的一种分布式 ID 生成算法。j-jdbc 的 DefaultKeyGenerator 是它的改进版。

java — DefaultKeyGenerator // 雪花 ID 的位分布（64位） // ┌──────────────┬──────────────┬──────────────┐ // │ 时间戳(41位) │ 工作节点(10位)│ 序列号(12位) │ // └──────────────┴──────────────┴──────────────┘ // 生成逻辑： return ((currentMillis - EPOCH) << TIMESTAMP_LEFT_SHIFT_BITS) | (workerId << WORKER_ID_LEFT_SHIFT_BITS) | sequence; // j-jdbc 的改进：奇偶间隔生成 // 在同一个毫秒内，序列号从上一个值继续递增 // 而不是每一毫秒都从 0 开始 // → 这在某些分库分表场景下性能更好
⏰ EPOCH = 2016-11-01

j-jdbc 的雪花 ID 以 2016年11月1日 为纪元起点。这意味着 ID 将从大约 41 位时间戳的中间值开始，保证在很长一段时间内不会溢出。

🧩 隐喻：酒店房卡系统

雪花 ID 就像一家连锁酒店的房卡系统。时间戳 = 入住日期（41 位日期码，可以覆盖 69 年）。工作节点 = 酒店分店编号（每个分店有自己的一盒房卡）。序列号 = 当天第几位客人。即使两个分店的客人同时入住，也绝不会拿到相同的房间号。

🗄️ 数据库 ID 生成器

除了雪花算法，j-jdbc-starter 模块还提供了一种基于数据库表的 ID 生成器——DbKeyGenerator。它适用于需要严格控制 ID 生成进度的场景。

java — DbKeyGenerator public class DbKeyGenerator implements KeyGenerator { // 内存中的 ID 缓存（每类一个计数器） private static final Map<String, AtomicLong> MAP = ...; public long generateKey(String type) { // 1. 第一次：从数据库读取种子值 (STEP << 4 = 16384) AtomicLong counter = MAP.computeIfAbsent(type, idType -> new AtomicLong(idHelper.get(idType))); // 2. 自增获取下一个 ID long result = counter.incrementAndGet(); // 3. 每消耗 1024 个 ID，批量更新数据库中的种子 if (result % IdHelper.STEP == 0) { execHelper.exec(() -> idHelper.update(type, result)); } return result; } }

关键是批量预分配：每次从数据库中取出一段连续的 ID（16384 个），在内存中分配。只有在用完了这一批之后才会再次访问数据库。这样既保证了 ID 的连续性，又减少了数据库压力。

♻️ 软删除 — 给数据第二次生命

软删除逻辑删除。数据不会物理删除，而是通过标记一个 deletedAt 字段表示"已删除"。查询时默认过滤删除的数据。可以随时恢复（undelete）。 意味着当你"删除"一条数据时，它并没有从数据库消失——而是在 deletedAt 字段上打了一个时间戳。这是企业级应用的标配做法。

java — BaseDO public abstract class BaseDO { @Column(order = 1003, comment = "删除时间") private Long deletedAt; // null 或 Constant.NO_DELETE = 未删除 public void forDelete() { deletedAt = System.currentTimeMillis(); // 👈 "删除"就是打个时间戳 } public void forUndelete() { deletedAt = Constant.NO_DELETE; // 👈 "恢复"就是重置为 0 } }

📢 DML 事件总线 — 监听每一次数据变更

DmlEventBus 是 j-jdbc 的通知系统。每次数据创建、更新、删除时，它都会通过 Guava EventBus 发布事件。你只需要注册监听器，就能在数据变更时获得通知。

java // 注册监听器： DmlEventBus.getEventBus(DmlType.CREATE) .register(new Object() { @Subscribe public void handle(UserDO user) { log.info("新用户创建了：{}", user.getName()); } }); // 内置事件点（BaseDaoImpl 中）： DmlEventBus.onCreate(t); // insert 后 DmlEventBus.onUpdate(t); // update 后 DmlEventBus.onDelete(t); // delete 后

📋 三种建表模式

j-jdbc-starter 模块中的 JdbcAutoConfiguration 提供了三种建表策略——通过一个配置项 j-jdbc.tableInit 控制。这是从开发到生产的进阶之路：

yml — application.yml # 模式 1: 仅创建不存在的表（开发环境推荐） j-jdbc: tableInit: 1 # 模式 2: 先删后建（测试环境快速重置） j-jdbc: tableInit: 2 # 模式 3: 自动添加缺少的列（生产环境安全升级） j-jdbc: tableInit: 3
☕ JdbcAutoConfiguration 三 种 模 式 对 比
// Mode 1 CREATE_TABLE

// Mode 2 DROP_CREATE_TABLE
表存在则删掉重建 → ⚠️ 会丢数据！仅测试用

模式 3 的原理非常巧妙： 它先执行 SELECT * FROM table WHERE 1=2（查询空结果），然后通过 JDBC 的 SqlRowSetMetaData.getColumnNames() 获取当前表的所有列名。接着与 TableInfo 中定义的列做对比——差集就是需要新加的列。整个过程不需要维护版本号或迁移脚本。

updateTable() — 安全演进

让我们深入 Mode 3 的 updateTable() 方法，它是生产环境中最有价值的特性：

java — BaseDaoImpl.updateTable() public int updateTable() { // 1. 查询空结果，获取当前表的元数据 SqlRowSet rowSet = template.queryForRowSet( selectNull().getSql(), ...); SqlRowSetMetaData md = rowSet.getMetaData(); // 2. 收集现有列名 Set<String> cn = new HashSet<>(); for (String columnName : md.getColumnNames()) { cn.add(columnName.toLowerCase()); } // 3. 找到"应有"但"没有"的列 List<ColumnInfo> columnInfos = columns.stream() .filter(c -> !cn.contains( c.getColumnName().toLowerCase())) .collect(Collectors.toList()); // 4. 逐列执行 ALTER TABLE ADD COLUMN for (ColumnInfo columnInfo : columnInfos) { SqlPrepared sql = sqlGenerator() .addColumn(columnInfo); template.update(sql.getSql(), sql.getParams()); } }

这解决了一个真实痛点：在传统开发中，每次新增字段都要写一个数据库迁移脚本。有了 j-jdbc，你只需要在 Java 类中添加一个新字段和 @Column 注解，重启应用，框架就会自动帮你 ALTER TABLE。

🔄 分库分表 — 应对海量数据

分库分表当单表数据量过大（>几千万行），将数据分散到多个数据库或多个表中存储。分片（Shard）是指按某种规则（如 ID 取模）决定数据放在哪个分片中。跨分片的聚合操作会更复杂。 是大型系统的必备技能。j-jdbc 通过 Apache ShardingSphere 的集成实现了这一点。

java — ModPreciseShardingAlgorithm public class ModPreciseShardingAlgorithm<T> implements PreciseShardingAlgorithm<T> { public String doSharding( Collection<String> targets, PreciseShardingValue<T> value) { // 取出 ID 中的数字部分 → 取模 → 决定去哪个分片 long va = StringUtil.getNumber( String.valueOf(value.getValue())); String suffix = StringUtil.getShardingSuffix( va % targets.size()); // 返回匹配的分片名，如 ds_0, ds_1 return targets.stream() .filter(t -> t.endsWith(suffix)) .findFirst() .orElseThrow(() -> new IllegalArgumentException()); } }

j-jdbc-sharding 模块支持四种数据源模式：

普通数据源 单库单表，最简单模式
分片数据源 多库 or 多表，ID 取模路由
主从数据源 一主多从，读写分离
分片+主从 分库分表 + 读写分离

🔍 出错时的诊断指南

作为直觉型编程者，遇到错误时知道 去哪里看、找什么，就是最实用的技能。以下是 j-jdbc 中常见的错误场景：

❌ "class UserDO is not set key"

你的 DO 类没有设置 @Key 主键。j-jdbc 要求必须有主键才能使用 find/findByIds 方法。在字段上添加 @Key 注解即可解决。

❌ "nothing to update"

你调用了 update() 但所有字段都是 null。j-jdbc 的智能 null 处理导致了"没有字段需要更新"。确保至少设置了一个非 null 字段，或者使用 updateNullFields() 主动声明要置空的字段。

❌ 主键类型错误

"@Key must be Long or String"——@Key 只支持 Long/long 和 String 类型。使用了 Integer 或其他类型会抛出 IllegalTypeException。

❌ 加密查询失败

如果你对某个 FIXED 加密字段用了 like 查询，会得到错误。在 where() 方法中，FIXED 加密只支持 eq/neq/in/notIn，不支持 like/between/gt/lt。设计上这是为了安全——加密字段不能做模糊匹配。

💡 AI 编码提示

当你的 AI 助手生成的代码遇到 "nothing to update" 错误时，告诉它："去检查 BaseDaoImpl.update() 方法中跳过 null 字段的逻辑，检查你传入的对象是否在调 update 前设置过字段值。使用 updateNullFields() 方法来处理需要置空的字段。"——这会让 AI 立刻意识到问题出在 null 处理上，而不是数据库连接。
