数据的旅程
CRUD 的生命之旅

现在你已经知道了 j-jdbc 用注解来描述数据。但当你调用 userDao.insert(user) 时——按下键盘到数据落库，中间发生了什么？

这一章我们将追踪一条数据从 Java 代码到数据库的完整旅程。你会发现 j-jdbc 的 SQL 生成器是真正的幕后英雄——它把简单的 Java 方法调用，翻译成精确的 SQL 语句。

🧩 隐喻：外卖配送

调用 userDao.insert(user) 就像在手机上点外卖。你（代码）只做了一件事——点了一下"下单"按钮。BaseDaoImpl 是外卖平台，接收你的订单。SqlGenerator 是厨房里的厨师，根据订单（注解信息）准备餐食（生成 SQL）。NamedParameterJdbcTemplate 是骑手，把餐食送到你的数据库门口。

一次查询的完整旅程

当你调用 userDao.find("U123@abc")，从方法调用到数据库查询结果返回，经历了这些步骤：

java — BaseDaoImpl.find() @Override public Optional<T> find(Object id) { // 1. 检查ID不为空 Preconditions.checkNotNull(id, "id is null"); // 2. 用 ConditionUtil 构建查询条件 List<Condition> conditions = ConditionUtil .newInstance() .field(key()).eq(id) .conditions(); // 3. 调用 queryField → SqlGenerator → 生成 SQL List<T> ts = queryField(conditions, null); // 4. 返回第一个结果 return ts.stream().findFirst(); }

ConditionUtil — 用 Java 写 WHERE 子句

ConditionUtil条件查询构建工具。提供链式调用 API 来构造 SQL 的 WHERE 条件，支持 eq、ne、gt、lt、gte、lte、like、in、notIn、between、empty 等操作。 是 j-jdbc 中最常用的工具之一。它让你用链式调用的方式构建查询条件，而不用手写 SQL。

☕ Java 条件代码 生 成 的 SQL
ConditionUtil.newInstance()
.field("age").gt(20).lte(89)
.field("nickname").like("%张三%")
.conditions()
WHERE age > 20 AND age <= 89
AND nickname LIKE '%张三%'

支持的所有条件操作：

.eq(val)等于 =
.ne(val)不等于 <>
.gt(val)大于 >
.lt(val)小于 <
.gte(val)大于等于 >=
.lte(val)小于等于 <=
.like(str)LIKE 模糊匹配
.in(list)IN 列表包含
.notIn(list)NOT IN 排除
.between(a,b)BETWEEN 范围
.empty(bool)IS NULL / IS NOT NULL

SQL 生成器 — Java 到 SQL 的自动翻译

j-jdbc 的 SQL 生成器体系由 SqlGenerator 接口定义，AbstractSqlGenerator 提供通用实现，SqlGenerator4Mysql 提供 MySQL 特化版本。这种设计让 j-jdbc 可以轻松支持其他数据库——只需要写一个新的实现类。

java — SqlGenerator 接口 public interface SqlGenerator<T> { SqlPrepared query(...); // SELECT 查询 SqlPrepared queryLimit(...); // 分页查询（带 LIMIT） SqlPrepared count(...); // SELECT COUNT(*) SqlPrepared insert(...); // INSERT SqlPrepared update(...); // UPDATE SqlPrepared delete(...); // DELETE List<SqlPrepared> createTableSql(); // CREATE TABLE SqlPrepared dropTableSql(); // DROP TABLE SqlPrepared addColumn(...); // ALTER TABLE ADD COLUMN String doc(); // 生成 Markdown 文档 }

看那个 where() 方法是如何将 Condition 对象翻译成 SQL 条件的——它遍历每个条件，检查哪个字段被设置了（eq、gt、like 等），然后拼装对应的 SQL 片段。这就像是一个精心设计的状态机！

🤯 啊哈！时刻

「参数化查询」是 j-jdbc 默认的安全保障： 所有的 SQL 参数都使用命名参数（如 :id__eq）而不是拼字符串。这意味着 SQL 注入攻击从一开始就被杜绝了——你不需要额外做任何安全配置。这是框架设计者的明智决定。

插入与更新 — 智能的 null 处理

j-jdbc 有一个非常贴心的设计：null 值不插入，null 值不更新。这意味着你可以只设置你想操作的字段，其他字段保持不变。

java — 智能 null // 插入时：只有设置了值的字段会出现在 INSERT 语句中 UserDO user = new UserDO(); user.setName("Alice"); user.setAge(25); // tags 为 null → 不会出现在 INSERT 中 userDao.insert(user); // → INSERT INTO DEMO_USER (id, name, age) VALUES (..., :name, :age) // 更新时：只有非 null 字段会更新 user.setAge(26); userDao.update(user); // → UPDATE DEMO_USER SET age=:age__set WHERE id=:id__eq // name 不会被更新！

更酷的是，如果你确实想把某个字段更新为 null，可以实现 updateNullFields() 方法告诉框架："这个字段我不是忘了设置，我是想把它清空。"
