隐藏的超能力
加密、ID 生成与安全删除

如果说前面三章是 j-jdbc 的"基础技能"，那这一章就是它的隐藏超能力。很多 CRUD 框架只能做增删改查，但 j-jdbc 内置了几个大多数开发者需要但懒得实现的特性——字段加密、分布式 ID 生成、逻辑删除和事件通知。

作为直觉型编程者，了解这些能力极其有价值：当你的 AI 助手建议在数据库中存储敏感数据时，你可以说"用 j-jdbc 的字段加密，别存明文"——这听起来就像你是个真正的后端架构师。

🔐 字段级加密 — 透明保护

字段加密在数据写入数据库时自动加密指定字段，读取时自动解密。对应用程序代码完全透明——你读写的是明文，数据库里存的是密文。 意味着你可以在 Java 代码中正常读写数据，但数据库里存的是密文。即使数据库被拖库，黑客看到的也是一堆乱码。

java // ===== FIXED 加密 ===== // 用固定密钥 "abcd" 加密。查询时框架自动加密比较。 @Column(encryptType = EncryptType.FIXED, encryptKey = "abcd") private String encryptTest; // ✓ 支持 eq, neq, in, notIn 查询 // ✗ 不支持 like 模糊查询 // ===== FIELD 加密 ===== // 用另一个字段 "id" 的值作为加密密钥 @Column(encryptType = EncryptType.FLIED, encryptKey = "id") private String name; // ✗ 每条记录密钥不同，不能作为查询条件 // ✓ 更安全 —— 泄露一条不影响其他数据
☕ 加密方式选择 什 么 时 候 用
EncryptType.FIXED
固定密钥：需要作为查询条件的字段（如手机号、邮箱），所有记录使用同一个密钥加密
EncryptType.FLIED
属性密钥：不需要查询的敏感字段（如密码、地址），每条记录使用不同的密钥（用其他字段的值加密）
EncryptType.NONE
不加密：默认选项

底层加密：AES 算法

j-jdbc 默认使用 AES（高级加密标准）来加密数据。AES 是目前最广泛使用的对称加密算法——从银行系统到 WhatsApp 都在用它。

java — Aes.java public class Aes implements Encryption { // 加密流程： public String encrypt(String content, String password) { // 1. 用密码生成加密密钥 KeyGenerator keyGen = KeyGenerator.getInstance("Aes"); SecureRandom random = SecureRandom .getInstance("SHA1PRNG"); random.setSeed(password.getBytes("utf-8")); keyGen.init(128, random); // 2. 初始化密码器（加密模式） Cipher cipher = Cipher.getInstance("Aes"); cipher.init(Cipher.ENCRYPT_MODE, secretKey); // 3. 加密 & Base64 编码 byte[] result = cipher.doFinal(content.getBytes("utf-8")); return Base64.getEncoder().encodeToString(result); } }

你也可以通过 EncryptionProvider 设置自己的加密算法——只需实现 Encryption 接口（encrypt/decrypt 两个方法），然后调用 EncryptionProvider.setEncryption(yourImpl)。

❄️ 雪花 ID — 分布式系统的身份证

雪花算法（Snowflake）Twitter 开源的一种分布式 ID 生成算法，生成 64 位的唯一整数。结构：时间戳（41位）+ 工作节点（10位）+ 序列号（12位）。可以在不依赖数据库的情况下生成全局唯一 ID。支持 1024 个节点，每毫秒单节点生成 4096 个 ID。 是 Twitter 发明的一种分布式 ID 生成算法。j-jdbc 的 DefaultKeyGenerator 是它的改进版。

java — DefaultKeyGenerator // 雪花 ID 的位分布（64位） // ┌──────────────┬──────────────┬──────────────┐ // │ 时间戳(41位) │ 工作节点(10位)│ 序列号(12位) │ // └──────────────┴──────────────┴──────────────┘ // 生成逻辑： return ((currentMillis - EPOCH) << TIMESTAMP_LEFT_SHIFT_BITS) | (workerId << WORKER_ID_LEFT_SHIFT_BITS) | sequence; // j-jdbc 的改进：奇偶间隔生成 // 在同一个毫秒内，序列号从上一个值继续递增 // 而不是每一毫秒都从 0 开始 // → 这在某些分库分表场景下性能更好
⏰ EPOCH = 2016-11-01

j-jdbc 的雪花 ID 以 2016年11月1日 为纪元起点。这意味着 ID 将从大约 41 位时间戳的中间值开始，保证在很长一段时间内不会溢出。

🧩 隐喻：酒店房卡系统

雪花 ID 就像一家连锁酒店的房卡系统。时间戳 = 入住日期（41 位日期码，可以覆盖 69 年）。工作节点 = 酒店分店编号（每个分店有自己的一盒房卡）。序列号 = 当天第几位客人。即使两个分店的客人同时入住，也绝不会拿到相同的房间号。

🗄️ 数据库 ID 生成器

除了雪花算法，j-jdbc-starter 模块还提供了一种基于数据库表的 ID 生成器——DbKeyGenerator。它适用于需要严格控制 ID 生成进度的场景。

java — DbKeyGenerator public class DbKeyGenerator implements KeyGenerator { // 内存中的 ID 缓存（每类一个计数器） private static final Map<String, AtomicLong> MAP = ...; public long generateKey(String type) { // 1. 第一次：从数据库读取种子值 (STEP << 4 = 16384) AtomicLong counter = MAP.computeIfAbsent(type, idType -> new AtomicLong(idHelper.get(idType))); // 2. 自增获取下一个 ID long result = counter.incrementAndGet(); // 3. 每消耗 1024 个 ID，批量更新数据库中的种子 if (result % IdHelper.STEP == 0) { execHelper.exec(() -> idHelper.update(type, result)); } return result; } }

关键是批量预分配：每次从数据库中取出一段连续的 ID（16384 个），在内存中分配。只有在用完了这一批之后才会再次访问数据库。这样既保证了 ID 的连续性，又减少了数据库压力。

♻️ 软删除 — 给数据第二次生命

软删除逻辑删除。数据不会物理删除，而是通过标记一个 deletedAt 字段表示"已删除"。查询时默认过滤删除的数据。可以随时恢复（undelete）。 意味着当你"删除"一条数据时，它并没有从数据库消失——而是在 deletedAt 字段上打了一个时间戳。这是企业级应用的标配做法。

java — BaseDO public abstract class BaseDO { @Column(order = 1003, comment = "删除时间") private Long deletedAt; // null 或 Constant.NO_DELETE = 未删除 public void forDelete() { deletedAt = System.currentTimeMillis(); // 👈 "删除"就是打个时间戳 } public void forUndelete() { deletedAt = Constant.NO_DELETE; // 👈 "恢复"就是重置为 0 } }

📢 DML 事件总线 — 监听每一次数据变更

DmlEventBus 是 j-jdbc 的通知系统。每次数据创建、更新、删除时，它都会通过 Guava EventBus 发布事件。你只需要注册监听器，就能在数据变更时获得通知。

java // 注册监听器： DmlEventBus.getEventBus(DmlType.CREATE) .register(new Object() { @Subscribe public void handle(UserDO user) { log.info("新用户创建了：{}", user.getName()); } }); // 内置事件点（BaseDaoImpl 中）： DmlEventBus.onCreate(t); // insert 后 DmlEventBus.onUpdate(t); // update 后 DmlEventBus.onDelete(t); // delete 后

❄
Snowflake（雪花） 我是纯内存计算，零数据库依赖！每毫秒能生成 4096 个唯一 ID，整个系统也不过 69 年才会用完。集群部署，无需加锁。
🗄
DbGenerator（数据库） 你说得对，但我的 ID 是连续且可预测的。适合做订单号、流水号这种需要直观排序的场景。而且我一次预取 16384 个 ID，数据库访问很少。
❄
连续？你依赖数据库，万一数据库挂了怎么办？我的时间戳+机器+序列结构保证了全局唯一且随时间递增，即使跨数据中心也不冲突。
🗄
我是 Starter 模块默认的——和 Spring Boot 配合使用，自动配置、自动建表。开箱即用不需要额外设置工作节点 ID。我们各有千秋！
⚙
KeyGeneratorProvider 好了好了，你们俩都是好样的。开发者可以通过我自由选择——KeyGeneratorProvider.setKeyGenerator(...)，喜欢谁就用谁！
