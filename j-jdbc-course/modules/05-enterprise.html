📋 三种建表模式

j-jdbc-starter 模块中的 JdbcAutoConfiguration 提供了三种建表策略——通过一个配置项 j-jdbc.tableInit 控制。这是从开发到生产的进阶之路：

yml — application.yml # 模式 1: 仅创建不存在的表（开发环境推荐） j-jdbc: tableInit: 1 # 模式 2: 先删后建（测试环境快速重置） j-jdbc: tableInit: 2 # 模式 3: 自动添加缺少的列（生产环境安全升级） j-jdbc: tableInit: 3
☕ JdbcAutoConfiguration 三 种 模 式 对 比
// Mode 1 CREATE_TABLE
表不存在才创建 → 安全、适合开发
// Mode 2 DROP_CREATE_TABLE
表存在则删掉重建 → ⚠️ 会丢数据！仅测试用
// Mode 3 UPDATE_TABLE
对比现有表和注解定义，自动添加缺失的列 → 生产环境升级表结构！
🤯 啊哈！时刻

模式 3 的原理非常巧妙： 它先执行 SELECT * FROM table WHERE 1=2（查询空结果），然后通过 JDBC 的 SqlRowSetMetaData.getColumnNames() 获取当前表的所有列名。接着与 TableInfo 中定义的列做对比——差集就是需要新加的列。整个过程不需要维护版本号或迁移脚本。

updateTable() — 安全演进

让我们深入 Mode 3 的 updateTable() 方法，它是生产环境中最有价值的特性：

java — BaseDaoImpl.updateTable() public int updateTable() { // 1. 查询空结果，获取当前表的元数据 SqlRowSet rowSet = template.queryForRowSet( selectNull().getSql(), ...); SqlRowSetMetaData md = rowSet.getMetaData(); // 2. 收集现有列名 Set<String> cn = new HashSet<>(); for (String columnName : md.getColumnNames()) { cn.add(columnName.toLowerCase()); } // 3. 找到"应有"但"没有"的列 List<ColumnInfo> columnInfos = columns.stream() .filter(c -> !cn.contains( c.getColumnName().toLowerCase())) .collect(Collectors.toList()); // 4. 逐列执行 ALTER TABLE ADD COLUMN for (ColumnInfo columnInfo : columnInfos) { SqlPrepared sql = sqlGenerator() .addColumn(columnInfo); template.update(sql.getSql(), sql.getParams()); } }

这解决了一个真实痛点：在传统开发中，每次新增字段都要写一个数据库迁移脚本。有了 j-jdbc，你只需要在 Java 类中添加一个新字段和 @Column 注解，重启应用，框架就会自动帮你 ALTER TABLE。

🔄 分库分表 — 应对海量数据

分库分表当单表数据量过大（>几千万行），将数据分散到多个数据库或多个表中存储。分片（Shard）是指按某种规则（如 ID 取模）决定数据放在哪个分片中。跨分片的聚合操作会更复杂。 是大型系统的必备技能。j-jdbc 通过 Apache ShardingSphere 的集成实现了这一点。

java — ModPreciseShardingAlgorithm public class ModPreciseShardingAlgorithm<T> implements PreciseShardingAlgorithm<T> { public String doSharding( Collection<String> targets, PreciseShardingValue<T> value) { // 取出 ID 中的数字部分 → 取模 → 决定去哪个分片 long va = StringUtil.getNumber( String.valueOf(value.getValue())); String suffix = StringUtil.getShardingSuffix( va % targets.size()); // 返回匹配的分片名，如 ds_0, ds_1 return targets.stream() .filter(t -> t.endsWith(suffix)) .findFirst() .orElseThrow(() -> new IllegalArgumentException()); } }

j-jdbc-sharding 模块支持四种数据源模式：

普通数据源 单库单表，最简单模式
分片数据源 多库 or 多表，ID 取模路由
主从数据源 一主多从，读写分离
分片+主从 分库分表 + 读写分离

🔍 出错时的诊断指南

作为直觉型编程者，遇到错误时知道 去哪里看、找什么，就是最实用的技能。以下是 j-jdbc 中常见的错误场景：

❌ "class UserDO is not set key"

你的 DO 类没有设置 @Key 主键。j-jdbc 要求必须有主键才能使用 find/findByIds 方法。在字段上添加 @Key 注解即可解决。

❌ "nothing to update"

你调用了 update() 但所有字段都是 null。j-jdbc 的智能 null 处理导致了"没有字段需要更新"。确保至少设置了一个非 null 字段，或者使用 updateNullFields() 主动声明要置空的字段。

❌ 主键类型错误

"@Key must be Long or String"——@Key 只支持 Long/long 和 String 类型。使用了 Integer 或其他类型会抛出 IllegalTypeException。

❌ 加密查询失败

如果你对某个 FIXED 加密字段用了 like 查询，会得到错误。在 where() 方法中，FIXED 加密只支持 eq/neq/in/notIn，不支持 like/between/gt/lt。设计上这是为了安全——加密字段不能做模糊匹配。

💡 AI 编码提示

当你的 AI 助手生成的代码遇到 "nothing to update" 错误时，告诉它："去检查 BaseDaoImpl.update() 方法中跳过 null 字段的逻辑，检查你传入的对象是否在调 update 前设置过字段值。使用 updateNullFields() 方法来处理需要置空的字段。"——这会让 AI 立刻意识到问题出在 null 处理上，而不是数据库连接。
